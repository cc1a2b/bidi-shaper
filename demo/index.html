A string is kept in the order you read it, and painted in the order you see it. Those are two different orders. bidi-shaper does the whole crossing — Arabic shaping, then UAX #9 reordering, then mirroring — and hands back a flat, visual-order string.

01 The pipeline

Three stations, one string

Type a line. It travels down through the two passes the library runs — shape() joins the Arabic letters, then render() reorders the runs and flips the brackets. Each tile keeps the index it had when it was stored, so you can watch where it lands once it is drawn.

03 The faults

Five faults, fixed before
the first glyph lands

Hand logical-order Arabic to a renderer that can’t shape or reorder and it gets every one of these wrong. bidi-shaper resolves all five in the string it returns.

  1. 01

    Cursive joining

    naive isolated, disconnected letters — ك ت ا ب

    render() joined to context — كتاب

  2. 02

    Reading direction

    naive runs come out reversed, end-first

    render() RTL runs flow right-to-left, LTR stays LTR

  3. 03

    Numbers inside RTL

    naive drift to the wrong end of the line

    render() stay left-to-right, in place — سنة ١٤٤٧

  4. 04

    Brackets & quotes

    naive ( ) point the wrong way

    render() mirrored on RTL runs — rule L4

  5. 05

    Lam + alef

    naive two stray letters — ل ا

    render() one ligature — ﻻ

Drop-in adapters المحوّلات

Structurally typed — they never import the host library, so they add nothing to your bundle. Same engine underneath.

jsPDFbidi-shaper/jspdf
import { rtlText } from "bidi-shaper/jspdf";

doc.text(rtlText("مرحبا بالعالم"), 20, 40);
PDFKitbidi-shaper/pdfkit
import { textBidi } from "bidi-shaper/pdfkit";

// features:[] so fontkit won't re-shape
textBidi(doc, "سلام دنیا", 72, 80);
Canvas 2Dbidi-shaper/canvas
import { fillTextBidi } from "bidi-shaper/canvas";

fillTextBidi(ctx, "قائمة (أ)", x, y);
